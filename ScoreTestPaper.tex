\section{Introduction}
\label{sec:introduction}

Model selection is an important step for
statistical analysis, allowing the practitioner to confirm the
reliability of their chosen model and any resulting
conclusions. Well-known model selection methods include comparing
models using likelihood ratio tests (LRTs), AIC values, and score
tests. Comparing two nested models via LRTs requires twice the
difference of each model's maximised log-likelihood
\parencite{neymanPearson}. Using the AIC means applying a penalty
reflecting model complexity to
each maximised log-likelihood, and comparing the results
\parencite{akaike1973}. To conduct a score test, a score test
statistic is calculated using log-likelihood gradients from the 
more complicated model -- specifically, the score
vector and information matrix \parencite{rao1948}. See
\textcite{rao2005} for an overview and recent developments of score
tests. 

Each method has its own advantages and
disadvantages. It is straightforward to perform the
calculations required to compare models via LRTs or AIC values. However, both
models must be fitted. To conduct score tests, only the simpler of the
two models needs to be fitted \parencite{rao2005}. However, the required
gradients can be challenging to obtain, often acting as a
deterrent for conducting these tests. 

Historically, LRTs and AIC values have been widely-used methods
of model selection. Although, when the models under comparison require
considerable time and resources to fit, these methods of comparison can
become inconvenient. This issue is amplified when considering
multiple potential models for a data set -- every single model in the
candidate set must be fitted, potentially rendering model selection
impractical.

In such cases, turning to score tests would be
beneficial. Fitting fewer models makes model selection more
practical. However, for usability, we require a straightforward way of
calculating the gradients required by the score test
statistic. Traditionally, options for obtaining these gradients
included symbolic differentiation and numeric approximations. The former
can quickly become unrealistic as model complexity grows, while the
latter often produces unstable and inaccurate
results \parencite{ADMBpaper}. In this paper, we demonstrate that
due to recent developments in statistical software, this issue is
solved -- it is now easier than
ever to implement score tests. Automatic differentiation (AD) tools 
allow for convenient procurement of the necessary gradients. 

Employing the chain rule of differentiation, AD tools produce exact
gradients of any specified function \parencite{ADMBpaper}. Using these tools often
only requires translating the desired function into an
appropriate coding language \parencite{TMBpaper}. For statistical models, these tools are
mainly used for model-fitting or calculating 
the Laplace approximation. Utilising the side-effects of these
capabilities -- repurposing their ability to calculate
exact log-likelihood gradients -- facilities a newly
convenient implementation of score tests.

This is especially valuable for random effects models, for example. Intractable
likelihoods involving potentially high 
dimensional integrals produce gradients with no closed form, resulting in
a host of challenges. Model fitting previously required tailored
software, even for relatively simple random effects models such as
Generalised Linear Mixed Models (GLMMs) \parencite{lme4-pkg}. However,
gradients remained elusive. \textcite{wang2022}
solved this issue for GLMMs, presenting extensive derivations
to obtain GLMM gradients. There remain many
classes of random effects models for which such solutions and
derivations don't exist and are impractical, rendering AD a key
advancement in the use of these models \parencite{skaug2002}. The ease
with which AD fits models and 
supplies gradients makes previously unfeasible tasks, such as model selection via score 
tests, viable in a way previously unseen. 

Therefore, to illustrate the benefits of score tests 
and AD tools, we will focus on the class of random effects models known as
spatial capture-recapture (SCR) models. Model selection is often difficult 
for SCR models. Likelihood equations for SCR models can be quite
sophisticated, involving products of two-dimensional spatial
integrals \parencite{borchersEfford2008}. Model-fitting, and therefore
using AIC values to compare 
models, is often resource intensive,
especially when many models are being considered. Due 
to the difficulties in symbolic differentiation or numerically 
approximating gradients, score tests have not generally been
used. Alternative approaches have been explored, but there are 
no well-established methods of model selection. Removing the barriers
to conducting score tests 
presents a new and potentially impactful solution to this issue. 

We will explore the newfound gains of conducting score tests using AD tools
via the framework introduced by \textcite{catchpoleMorgan1996}, and
later developed by \textcite{mccreaMorgan2011}. Following the
convention of organising the
candidate set into a tree structure, the framework provides an efficient way of
navigating through the set using score tests, 
amplifying the benefits of score tests for model selection. The issue
of choosing a model from a   
potentially vast candidate set of models of varying complexity is reduced to the
task of calculating score statistics and fitting a few, potentially
relatively simple, models. 

We present our findings using an illustrative example. We will employ
the `RTMB' R package -- a widely-used AD tool -- 
to implement score tests for SCR model selection using
the ovenbird SCR data set, available in the `secr' R
package \parencite{secr-pkg}. Compared to traditional methods, we examine how our
approach enhances the convenience and accessibility of
model selection. By considering a range of candidate models, we also
explore how to accommodate models that aren't
nested, a potentially common issue. As with the possible difficulties of model
selection, our insights are not specific to any one field of
research or class of model. They may be applied to a vast range of models.

\section{Illustrative data set: ovenbirds}
\label{sec:ovenbirds}

In \textcite{borchersEfford2008}, alongside introducing Maximum
Likelihood methods for SCR models,
the authors illustrate a potential approach 
for SCR model selection. The data set used consists of captures of birds
(red-eyed vireos) from a mist-netting survey on the Patuxent Research Refuge 
in Maryland, USA, conducted by C.S. Robbins from 1961 to 1972. A variety of SCR
models are fitted and the final model is selected via AIC.

For easy comparison to the standard set by such a well-known example, we
work with a similar data set. The `ovenbird' data
set, available via the `secr' R package, contains adult captures from a
mist-netting survey of ovenbirds conducted from 2005 to 2009 at the Patuxent
Research Refuge by D. K. Dawson and M. G. Efford. As
described in the documentation, the forest in the area has
changed little since the previous survey conducted by C.S. Robbins
\parencite{secr-pkg}. One bird was killed by a predator while in a net
in 2009 \parencite{secr-pkg}, and for our analysis we remove this loss
from the data set. We retain any 
records of individual birds being captured more than once in a
day. 

Forty four mist nets were passively operated for nine or
ten non-consecutive days in late May and June of each year. In the
context of SCR, populations separated by time (e.g. 
years) are referred to as `sessions'
\parencite{effordBook2009}. Therefore, our 
data set contains five sessions, each with nine or ten sampling occasions. As
is standard when fitting multi-session SCR models, we each session as
if from an independent closed population
\parencite{effordBook2009}. Using closed populations means we do not consider any changes in
population size each year due to births, deaths, etc. 

Our candidate model set contains all SCR models that share parameter
dependencies considered in
\textcite{borchersEfford2008}. The latter candidate set contains
fifteen models. By considering all possible combinations of the
parameter dependencies presented in \textcite{borchersEfford2008}, our
set contains forty five models.

To construct an accurate picture of the benefits afforded by model
selection via score tests and AD tools, we implement model selection
via AIC values alongside our proposed approach. We compare a new, more efficient,
approach to model selection with a widely-used traditional approach,
in the context of an example that would be familiar to practitioners.

describes animal density, while $g_0$ and $\sigma$ describe

In Table \ref{table:dependencies} (see \textcite{secr-book} for further detail):
\begin{itemize}
  \item `$\sim 1$' represents a parameter being held pooled across
    sessions. For example, with $D \sim 1$, we have only one
    estimate of $D$ and use it for all sessions;
  \item `$\sim \text{session}$' means a parameter has session-specific
    levels. If we specify $D \sim \text{session}$, we have a separate estimate
    of $D$ for each session;
  \item `$\sim \text{Session}$' represents a trend across
    sessions. With `$D \sim \text{Session}$', we  have $D(y) =
    exp(\beta_0 + \beta_1 y)$, 
    where $y$ is the session number \parencite{borchersEfford2008}. In
    our case, $y=1,...,5$;
  \item `$\sim \text{h2}$' denotes a
    two-class finite mixture model. For example, with `$\sigma \sim \text{h2}$', we assume
    there are two groups in the population: Group 1 uses $\sigma_1$,
    while Group 2 uses $\sigma_2$. An 
    individual falls in Group 1 with probability $p$ and in Group 2
    with probability $(1-p)$. These estimates are pooled across sessions; 
  \item `$\sim \text{b}$' represents a trap response. With $g_0 \sim
    \text{b}$, we have two estimates of $g_0$: one for individuals
    captured on previously in a session, and one for individuals
    that have not yet been captured in a session. These estimates are pooled
    across sessions;
  \item `$\sim \text{session} + \text{b}$' means we have
    session-specific levels and a trap response -- e.g. with $g_0 \sim \text{session} +
    \text{b}$, we fit a model with two separate estimates of $g_0$ for
    each session: one for previously captured individuals, and one for
    individuals that have not been caught yet. 
\end{itemize}

Using score tests via RTMB greatly
enhances the convenience of model selection for a comprehensive
candidate set like ours. SCR models do not fit the description of a
`standard' random effects model. Evaluating
the likelihood of each model in the candidate set requires a
product of two-dimensional integrals, where each integral is an infinite
sum over the random effects in the model -- the captured animals' home
range centers \parencite{borchersEfford2008} -- 
adding a layer of complexity that further amplifies the challenges of working with a
sophisticated class of models. Model fitting becomes resource
intensive, hindering the utility of traditional model selection
approaches such as LRTs and AIC values.

Regardless, model selection via AIC values remains a common approach
for SCR model selection. Although SCR models are useful for estimating
animal abundance and therefore popular in the field of statistical
ecology, these models aren't widely used enough for there to be
established methods for finding derivatives or tailored approaches for
convenient model selection. As we see below, our general use,
straightforward method significantly enhances the
accessibility of model selection with little cost.  

\subsection{Calculating score test statistics with RTMB}
\label{sec:calculating-stats-rtmb}

Texts such as \textcite{rao2005} provide detailed information on score
tests. We provide a brief overview. When comparing two nested models
using a score test, our null hypothesis 
is that the simpler model is adequate. Therefore, we will refer to the
simpler model as the `$H_0$ model', and the larger model as the `$H_1$
model'. We label the parameter vector for the $H_0$ model as $\thetaZero$,
and the parameter vector for the $H_1$ model as $\thetaOne$. The
steps are as follows:
\begin{enumerate}
\item Fit the $H_0$ model, extract the Maximum Likelihood Estiamte
  (MLE) vector $\thetaZeroMLE$; 
\item Translate $\thetaZeroMLE$ into a $\thetaOne$ vector --
  i.e. identify a $\thetaOne$ vector that shares the constraints of
  the $H_0$ model and uses the estimates of $\thetaZeroMLE$. We
  label the resulting vector $\thetaOneTilde$. Plugging
  $\thetaOneTilde$ into the $H_1$ model returns the $H_0$ model;
\item Calculate:
  \begin{equation}
    \boldsymbol{u}_1 (\thetaOneTilde)^T \mathcal{I}_1
    (\thetaOneTilde) \boldsymbol{u}_1 (\thetaOneTilde)
  \end{equation}
  where $\boldsymbol{u}_1 (\thetaOneTilde)$ is the score vector of
  the $H_1$ model evaluated at $\thetaOneTilde$, and $\mathcal{I}_1
  (\thetaOneTilde)$ is the information matrix of the $H_1$ model
  evaluated at the same. This gives the score test
  statistic;
\item Compare this statistic to a Chi-squared distribution where
  the degrees of freedom is equal to the difference in the number
  of parameters for the $H_0$ and $H_1$ model. 
\end{enumerate}

Note that we focus on the use of the observed information matrix
in Step 3. See section (\textbf{insert number}) of the Discussion for
more details. Intuitively, the 
formula in Step 3 means the score test statistic summarises the
distance between $\thetaOneTilde$ and the peak of the $H_1$
log-likelihood. That is, it summarises the distance between
$\thetaOneTilde$ and the unconstrained $H_1$ model
MLE vector, $\thetaOneMLE$. If this distance is large, the
difference in fit between the $H_0$ and $H_1$ model is likely
significant enough for the $H_1$ model to be preferable despite its
added complexity, and vice versa.

By giving the gradient of the $H_1$
log-likelihood at $\thetaOneTilde$, $\boldsymbol{u}_1
(\thetaOneTilde)$ gives a rough idea of the distance between
$\thetaOneTilde$ and $\thetaOneMLE$. Additionally considering the curvature
via $\mathcal{I}_1 (\thetaOneTilde)$ creates an
accurate picture of this distance. For example, as we take the inverse
of the information matrix, low
curvature at $\thetaOneTilde$ means
the score test statistic becomes inflated. A heavier penalty is placed on the
gradient, as even a small gradient could relate to a large distance
between $\thetaOneTilde$ and $\thetaOneMLE$.

Previously, the obstacle in implementing score tests was obtaining the
score vector and information matrix. Using recent software advancements, we employ
the `RTMB' R package -- a commonly-used AD tool -- to implement this. By
writing the log-likelihoods for all $H_1$ models of interest
as R functions, it is straightforward to extract the score vectors
and information matrices we need. Note that due to the speed and
accuracy of AD, we also use RTMB to fit our $H_0$ models. Refer to
section \ref{sec:score-test-example} of the Appendix for a simple
illustrative example of implementing a score test using RTMB. 

\subsection{Implementing the score test framework}
\label{sec:implementing-framework}

To implement the model selection framework of
\textcite{catchpoleMorgan1996}, the candidate set is organised 
into levels and, beginning with the simplest models, score tests
define a path through these levels to select the
final model. We begin by identifying the 
most basic model. Using the terminology of \textcite{mccreaMorgan2011},
this model makes up Level 0 of the candidate set. Level 1 contains all models
with one added parameter dependency. 

We conduct score tests to
compare all Level 1 models to the Level 0 model, with the null hypothesis
that the Level 0 model is adequate for the data. Applying the ideas of
Section \ref{sec:calculating-stats-rtmb}, we refer to the Level 0
model as the `$H_0$ model', and all of the Level 1 models as `$H_1$
models'. The $H_1$ model for the 
test with the smallest significant 
p-value defines the new null hypothesis -- that is, it becomes the new
$H_0$ model.

Level 2 is then constructed, consisting of all models with one
additional parameter dependency relative to this model, and the
process is repeated. We stop when none of the score tests produce
significant p-values. At this point, the $H_0$ model is the final
chosen model. 

As seen in Section \ref{sec:calculating-stats-rtmb}, when conducting
each score test, only the $H_0$ model needs to be fitted. And the stepwise nature of the 
framework enhances the convenience of any model-fitting -- the MLEs
from a given $H_0$ model may provide appropriate starting values for
fitting the next $H_0$ model \parencite{mccreaMorgan2011}. By
harnessing the utility of score tests and the convenience afforded by
AD, model selection with this framework now involves accessible
calculations, coupled with the likely straightforward fitting of few models. 

\subsection{Issues with nesting}
\label{sec:nesting-issues}

When working with comprehensive candidate sets, issues with nesting
may arise. There may be models in the candidate set that do not 
fit into the levels described above. We extend the framework of
\textcite{catchpoleMorgan1996} to accommodate this.

For models to correctly fit into the levels of our chosen model
selection framework, they must be nested appropriately for score
tests. There must be only one choice of $\thetaOneTilde$. This ensures
any changes in $\thetaOneTilde$ cause a change in the $H_1$
log-likelihood, resulting in the required curvature at
$\thetaOneTilde$ for the score test statistic to inform how far
$\thetaOneTilde$ is from the peak.

This may be violated if our candidate set contains
models with different parameters. Where issues arise, we propose extending
the framework of \textcite{catchpoleMorgan1996} by separating the
candidate set into groups of models with the same parameters and
implementing the model selection framework on each group. The chosen
models can be compared using AIC values to select the 
final model. Being a well-established and straightforward method
for comparing a few models, using AIC values facilitates our extension
of the chosen framework, allowing practitioners to harness the
convenience of score tests even when issues with nesting arise.

\subsection{SCR overview and candidate set}
\label{sec:overview-candidate-models}

Using records of when and where individuals were captured, SCR models
allow for estimation of animal density and distribution
\parencite{borchersEfford2008}. As described in \textcite{secr-book},
there are two parts to an SCR model: the state model and the
observation model. The state model describes the distribution of the animal population,
the `biological reality we wish to describe'. The observation model
describes the sampling process we use to understand this
population. Selecting appropriate state and observation models that accurately
reflect the population and survey enhance the validity of
inference. For understanding, we briefly describe our selections below. Please refer to
texts such as \textcite{secr-book} for further details.

Specifying the state model means choosing a spatial point process to
describe the distribution of animals in the survey area
\parencite{borchersFewster2016}. It is common to select a spatial
Poisson process \parencite{secr-book}. Specifically, following the
example of \parencite{borchersEfford2008}, we use a
homogeneous Poisson process, so animals are assumed to be
distributed uniformly across the survey area. Using the state model,
we estimate the parameter $D$, animal density in the area of interest.

To specify the observation process, we select a detection function for
our SCR models. This describes how an animal's probability of detection
changes as a function of the distance between their activity centre and a
detector. An animal's activity centre is the centre of their home
range \parencite{secr-book}.

The choice of detection function dictates the `detection' parameters
to be estimated. Although \textcite{borchersEfford2008} consider
SCR models with half-normal and hazard rate detection functions, we only consider models with a
half-normal detection function. Therefore, our detection
parameters are $g_0$ and $\sigma$. Hazard rate detection
functions are seldom used and generally not recommended due to their
long tail \parencite{secr-pkg}.

Refer to texts such as \textcite{secr-book} for further details on
state models and observation models. 

When fitting a SCR model, along with the
state/observation models, it is standard to specify any dependencies for $D$
and the detection parameters. Following the example set by
\textcite{borchersEfford2008}, we 
consider models with the dependencies shown in Table
\ref{table:dependencies} (see Section \ref{sec:candidate-set}).
